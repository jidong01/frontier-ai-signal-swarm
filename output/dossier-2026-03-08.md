## 심층 탐구 1: 에이전트가 AI를 만든다 — Karpathy Autoresearch

### 무슨 일이 일어났는가

Andrej Karpathy가 2026년 3월 6일, GitHub에 `autoresearch`를 공개했다[^1]. 지금까지 그의 교육 시리즈(micrograd → makemore → nanoGPT → microgpt)가 "인간이 AI를 이해하도록 돕는" 것이었다면, autoresearch는 방향이 역전된다 — 에이전트가 AI 연구를 수행한다.

시스템의 구조는 놀랍도록 단순하다: 단일 GPU 위에서 실행되는 소형 LLM 훈련 셋업(`nanochat`)에 AI 에이전트를 붙여주고, 에이전트는 밤새 자율적으로 실험을 반복한다. 5분간 훈련 → 이전과 비교 → 개선됐으면 유지, 아니면 롤백. 사용자가 아침에 일어나면 실험 로그와 (아마도) 더 나은 모델이 기다리고 있다.

가장 흥미로운 설계 결정은 `program.md`다. 에이전트는 Python 코드를 직접 수정하지 않는다. 대신 `program.md`라는 Markdown 파일이 연구 방향, 가설, 탐색 공간을 정의하고, 에이전트는 이 "프로그램"을 읽고 실험을 설계한다. Karpathy가 README에서 이렇게 설명한다: "당신은 Python 파일을 건드리지 않는다. 대신 에이전트에게 컨텍스트를 제공하는 `program.md` Markdown 파일을 프로그래밍한다."

### 왜 이것이 흥미로운가

2973개의 GitHub 스타가 18시간 만에 붙었다. 반응의 속도도 인상적이지만, 이 프로젝트가 촉발하는 질문이 더 흥미롭다: **에이전트를 만드는 사람의 역할이 무엇인가?**

Karpathy의 구조에서 인간의 역할은 세 가지로 압축된다:
1. 연구 방향 설정 (`program.md` 작성)
2. 실험 결과 검토
3. 방향 수정

코드를 쓰지 않는다. 가설을 테스트하지 않는다. 파라미터를 바꾸지 않는다. 에이전트가 한다.

이것은 메타 하네스(meta-harness) 개념의 처음으로 공개된 간결한 구현이다. 하네스가 에이전트를 사용해 특정 도메인 태스크를 수행한다면, 메타 하네스는 에이전트를 사용해 **에이전트 자체를 개선**한다. autoresearch에서 `program.md`는 하네스의 최상위 레이어다 — 모델 아키텍처를 탐색하는 에이전트에게 어떤 방향으로 탐색할지를 알려주는 지도.

### 실제로 어떻게 작동하는가

autoresearch의 실행 루프를 분해하면:

```
[program.md 읽기]
      ↓
[현재 기준점(baseline) 5분 훈련]
      ↓
[에이전트: "다음 실험 설계" — nanochat 코드 수정 제안]
      ↓
[수정된 코드 적용, 5분 훈련]
      ↓
[이전 결과와 비교 (validation loss)]
      ↓
[개선됐으면 새 기준점으로 채택, 아니면 롤백]
      ↓
[반복 — 아침까지]
```

여기서 "훈련 시간 5분"이라는 제약이 핵심이다. 에이전트가 더 좋은 아키텍처를 탐색하려면 빠른 피드백 루프가 필요하다. 5분은 실험이 충분히 의미 있는 신호를 주면서도, 에이전트가 하룻밤에 수십 번 반복할 수 있는 시간이다. 이것은 연구 루프의 주기를 "주"에서 "시간"으로 단축하는 압축이다.

한 가지 흥미로운 미결 문제가 이슈 트래커에 이미 등장했다: `#22 "Low creativity"` — 에이전트가 탐색 공간의 비슷한 영역을 반복해서 시도한다는 것[^2]. 이것은 탐색-활용 트레이드오프(Exploration-Exploitation Tradeoff)의 에이전트 버전이다. 에이전트가 "이미 효과를 본 것"을 반복하는 경향이 있다면, `program.md`에 의도적인 다양성 주입 메커니즘이 필요하다.

### 이것이 바꾸는 것

빌더의 관점에서 autoresearch가 제기하는 질문은 이것이다: **내가 지금 하고 있는 반복적 실험 중 어느 부분을 에이전트에게 위임할 수 있는가?**

"에이전트에게 실험을 맡긴다"는 것의 선행 조건을 autoresearch가 명확히 드러낸다:
- 실험의 사이클 타임이 짧아야 한다 (여기선 5분 훈련)
- 결과의 품질을 측정하는 신호가 명확해야 한다 (여기선 validation loss)
- 탐색 공간이 `program.md` 수준으로 명세 가능해야 한다

이 세 조건이 갖춰지면 에이전트는 연구자보다 탐색 속도가 빠르다. 조건이 없으면 에이전트는 방향을 잃고 창의성 없이 같은 자리를 맴돈다.

### 아직 모르는 것

- 에이전트가 정말 "새로운" 아이디어를 만드는가, 아니면 훈련 데이터에서 본 패턴을 조합하는가?
- `program.md`의 품질이 얼마나 결과에 영향을 주는가 — 이것은 새로운 종류의 프롬프트 엔지니어링인가?
- 수렴하는 기준점의 품질 상한선은 무엇이 결정하는가? (단일 GPU 제약, 훈련 데이터, 모델 아키텍처 탐색 공간 중 어느 것이 병목인가?)

### 더 알고 싶다면

- GitHub `karpathy/autoresearch` — README와 issues가 이미 좋은 토론의 시작점
- Karpathy의 이전 교육 시리즈: micrograd → makemore → nanoGPT → microgpt의 흐름을 이해하면 autoresearch의 위치가 명확해진다

## 심층 탐구 2: 시작 버튼이 사라진 에이전트

### 무슨 일이 일어났는가

Cursor가 2026년 3월 5일 Automations를 출시했다[^3][^4]. 이름이 단순하지만, 이것이 의미하는 것은 단순하지 않다: **에이전트가 더 이상 인간이 시작 버튼을 누르기를 기다리지 않는다.**

트리거 목록이 이 변화를 잘 보여준다:
- 새로운 GitHub Pull Request → 에이전트가 자동으로 코드 리뷰
- Slack 메시지 → 에이전트가 관련 코드를 조사
- PagerDuty 알람 → 에이전트가 로그 분석하고 패치 제안
- 타이머 (크론) → 에이전트가 정기 작업 실행

현재 Cursor는 플랫폼 전체에서 시간당 수백 건의 Automation을 실행하고 있다. Cursor CEO Michael Truell은 이 변화를 이렇게 설명했다: "인간이 항상 시작하는 것이 아니라, 적절한 시점에 호출받는다."

### 왜 이것이 흥미로운가

에이전트 AI 도구의 진화를 세 단계로 나눌 수 있다:

| 단계 | 패러다임 | 인간의 역할 |
|------|---------|------------|
| 1단계 (2022-2024) | 자동완성 | 매 줄마다 수락/거부 |
| 2단계 (2024-2025) | 채팅/에이전트 | 프롬프트 입력, 결과 감독 |
| **3단계 (2026-)** | **이벤트 기반** | **정책 설정, 예외 시 호출** |

3단계에서 변하는 것은 **인간의 기본 자세**다. 1-2단계에서 인간은 능동적으로 에이전트를 시작하고 결과를 검토한다. 3단계에서 인간은 에이전트가 자율로 작동하도록 허용하고, 에이전트가 예외를 만나거나 확인이 필요할 때 호출받는다. 에이전트가 주도권을 갖는 것이다.

이것은 추상적 주장이 아니다. Cursor의 Bugbot이 구체적 사례다: Bugbot은 매 커밋마다 자동으로 코드를 분석하고, 버그를 발견하면 Slack에 보고한다. 개발자가 Cursor를 열어 "이 커밋을 리뷰해줘"라고 요청하는 것이 아니다 — Bugbot이 이미 해놓는다.

### 이벤트 기반 에이전트의 설계 문제

이 패러다임이 에이전트를 만드는 사람에게 제기하는 새로운 문제들:

**1. 트리거 설계:** 어떤 이벤트가 에이전트를 깨워야 하는가? 너무 많은 이벤트를 트리거로 쓰면 에이전트가 과부하되고, 너무 적으면 자동화의 가치가 없다. 트리거 선택은 기술 문제가 아니라 비즈니스 판단 문제다.

**2. 종료 조건:** 이벤트 기반 에이전트는 "언제 멈춰야 하는가"가 불명확하다. 배치 에이전트는 태스크가 완료되면 끝난다. 이벤트 기반 에이전트는 계속 새로운 이벤트를 기다린다. 이것은 에이전트의 "생명 주기 관리"라는 새로운 문제를 만든다.

**3. 에스컬레이션 정책:** 에이전트가 확신이 없을 때 인간을 호출하는 기준은 무엇인가? 호출을 너무 자주 하면 "항상 켜진 에이전트"의 가치가 없고, 너무 드물게 하면 에이전트가 잘못된 방향으로 진행한다. 이것은 에이전트의 "자율성 임계값(Autonomy Threshold)" 설정 문제다.

**4. 에이전트 간 충돌:** 같은 코드베이스에 여러 Automation이 설정되면 에이전트들이 서로 충돌할 수 있다. 에이전트 A가 파일을 수정하는 동안 에이전트 B가 같은 파일을 리뷰하려면? 이것은 에이전트 조율 문제다.

Cursor의 Automations가 이 문제들을 어떻게 해결했는지는 아직 공개 정보가 제한적이다. 하지만 문제를 제기했다는 것 자체가 중요하다 — 이 아키텍처 결정들이 다음 세대 에이전트 인프라의 설계 기준이 될 것이다.

### 동시에 일어나는 일: 프로토콜의 전쟁

이벤트 기반 에이전트가 현실이 되면서, 에이전트들이 서로 통신하는 방법의 표준화가 급해졌다. 현재 경쟁하는 프로토콜들[^5][^6]:

| 프로토콜 | 개발사 | 레이어 | 역할 |
|---------|------|--------|------|
| **MCP** (Model Context Protocol) | Anthropic | 에이전트 ↔ 도구 | AI가 외부 도구/데이터에 접근 |
| **A2A** (Agent-to-Agent) | Google DeepMind | 에이전트 ↔ 에이전트 | 에이전트 간 태스크 위임 |
| **ACP** (Agent Communication Protocol) | IBM/Linux Foundation | 에이전트 ↔ 에이전트 | 엔터프라이즈 에이전트 통신 |
| **ANP** (Agent Network Protocol) | 분산 프로젝트 | 에이전트 발견 | 에이전트의 공개 디렉터리 |
| **AG-UI** | 커뮤니티 | 에이전트 ↔ UI | 에이전트 결과의 UI 표준화 |

이 프로토콜들이 동시에 등장한 것은 우연이 아니다. Cursor Automations처럼 에이전트가 실제로 이벤트를 처리하고 다른 에이전트에게 태스크를 넘기기 시작하면, 그 통신 방식이 표준화되어야 한다.

MCP와 A2A를 잘못 이해하는 흔한 오류가 있다: 둘이 경쟁한다는 것. 실제로는 레이어가 다르다. MCP는 에이전트가 데이터베이스, GitHub, Slack에 접근하는 방법이고, A2A는 에이전트가 다른 에이전트에게 서브태스크를 위임하는 방법이다. 마치 HTTP(인간 ↔ 웹서버)와 gRPC(서버 ↔ 서버)가 경쟁하지 않는 것처럼.

그러나 ACP, ANP, AG-UI까지 등장하면 그림이 복잡해진다. "에이전트 인터넷"의 TCP/IP 모멘트가 오고 있다고 부르는 사람들이 있는데[^7], 비유는 매혹적이지만 한 가지 차이가 있다 — TCP/IP는 단 하나였다. 현재는 5개 이상의 프로토콜이 서로 다른 이해관계를 가진 대기업들(Anthropic, Google, Microsoft, IBM)이 밀고 있다. 어떤 것이 살아남을지는 아직 아무도 모른다. 채택이 느린 자발적 표준의 역사를 보면, 생태계 규모가 가장 큰 것이 이긴다 — 그리고 현재 MCP는 97M+ 다운로드로 가장 앞서 있다.

### 에이전트 생태계의 오늘

Kalinowski의 OpenAI 사임이 이 맥락에서 의미 있다[^8]. 그는 "거버넌스 우려(governance concern)"를 이유로 들었다 — "발표가 가드레일 없이 급하게 진행됐다"는 것. 에이전트가 이벤트에 반응하고, 코드를 수정하고, 결정을 내리는 시대에 거버넌스 — 에이전트가 무엇을 허용되고 무엇이 금지되는지, 누가 그 경계를 설정하는지 — 는 기술 문제가 아니라 조직 문제다. 그리고 최상위 기술 임원이 이 문제로 회사를 떠나는 것은, 에이전트 시대의 거버넌스 위기가 아직 해결되지 않았음을 보여준다.

## 오늘의 개념 도구

**자율성 임계값 (Autonomy Threshold)**
에이전트가 인간 개입 없이 진행할 수 있는 불확실성의 최대 허용 수준. 이것을 낮게 설정하면 에이전트가 모든 것을 확인받으러 오고, 높게 설정하면 에이전트가 혼자 잘못된 방향으로 진행한다. 이벤트 기반 에이전트 시스템에서는 이 값을 태스크 유형별로 다르게 설정해야 한다.

**연구 루프의 압축 (Research Loop Compression)**
autoresearch에서 드러나는 패턴. 인간 연구자의 실험 사이클(가설 → 구현 → 실험 → 분석 = 수 일)을 에이전트가 분 단위로 압축할 때, 병목은 아이디어의 속도가 아니라 빠른 피드백 신호의 존재 여부로 이동한다.

**이벤트 기반 하네스 (Event-Driven Harness)**
인간의 명시적 요청(prompt)을 시작 트리거로 쓰는 대신, 외부 이벤트(GitHub 이벤트, 알람, 타이머)를 시작 트리거로 쓰는 하네스 아키텍처. 인간의 역할이 "에이전트를 시작하는 사람"에서 "정책을 설정하고 예외에 개입하는 사람"으로 전환된다.

**프로토콜 레이어 분리 (Protocol Layer Separation)**
MCP(에이전트↔도구), A2A(에이전트↔에이전트), WebMCP(에이전트↔브라우저)가 각각 다른 통신 문제를 해결한다는 인식. 이 셋을 혼동하면 아키텍처 설계에서 잘못된 레이어에 해결책을 적용하게 된다.

---

## 시그널 간 연결

오늘의 세 시그널은 같은 방향을 가리킨다: **에이전트 인프라의 성숙.**

Karpathy autoresearch는 에이전트가 할 수 있는 것의 상한선을 보여준다 — 에이전트가 스스로 더 나은 에이전트를 만들 수 있다. Cursor Automations는 에이전트가 실제로 대규모 프로덕션에서 이벤트 기반으로 작동하는 첫 번째 증거다. 에이전트 프로토콜의 등장은 이 에이전트들이 서로 통신하는 인프라 레이어가 형성되고 있음을 보여준다.

이 세 변화가 동시에 일어나는 것은 우연이 아니다. 에이전트 AI의 첫 세대는 "모델이 좋아지면 에이전트가 더 잘 한다"는 단순한 가정 위에 있었다. 두 번째 세대는 다른 교훈을 배우고 있다: 에이전트의 한계는 주로 아키텍처에서 온다. 어떻게 시작되는가, 어떻게 통신하는가, 어떻게 중단되는가, 어떻게 인간에게 에스컬레이션하는가.

이것이 하네스 엔지니어링의 다음 장이다 — 단일 에이전트 루프에서 이벤트 기반 에이전트 네트워크로.

## 열린 질문들

1. **에이전트가 AI 연구를 한다면, 연구자의 역할은 무엇인가?** autoresearch의 `program.md`를 쓰는 것은 "연구를 한다"는 것인가, "연구를 설계한다"는 것인가? 이 구분이 중요한가?

2. **이벤트 기반 에이전트의 거버넌스는 누가 설정하는가?** Cursor Automations에서 어떤 이벤트가 에이전트를 트리거해야 하는지를 결정하는 사람은 개발자인가, 팀 리더인가, 법무팀인가? Kalinowski의 사임이 암시하듯, 이 질문은 기술을 만드는 회사 안에서도 아직 합의되지 않았다.

3. **프로토콜 전쟁에서 빌더는 어떤 것에 지금 투자해야 하는가?** MCP가 97M 다운로드로 앞서지만, A2A는 Google의 지원을 받는다. 스타트업이 MCP 위에서만 빌드하면, 나중에 A2A로 전환 비용이 얼마나 될까?

4. **자율 연구 에이전트의 성능 상한선은 어디인가?** autoresearch의 에이전트는 단일 GPU에서 nanochat이라는 제한된 모델을 개선한다. 이 접근법이 프론티어 모델 훈련에 적용될 때 무엇이 달라지는가?

5. **이벤트 기반 에이전트가 "항상 켜져" 있다는 것이 감사(audit)와 책임(accountability)에 어떤 영향을 주는가?** 에이전트가 자율로 코드를 수정했을 때, 버그가 발생하면 누구의 책임인가?

## 오늘의 학습 포인트

1. **에이전트 입력 모델을 재설계해야 한다.** "사용자가 프롬프트를 입력한다"는 가정을 버리고, "어떤 이벤트가 이 에이전트를 트리거해야 하는가"를 설계의 첫 질문으로 바꿔야 한다.

2. **빠른 피드백 신호가 자율 개선의 전제 조건이다.** autoresearch가 작동하는 이유는 5분 훈련이라는 빠른 피드백 루프 때문이다. 자신의 도메인에서 이에 해당하는 "5분 피드백"이 무엇인지 찾는 것이 메타 에이전트 설계의 첫 단계다.

3. **MCP와 A2A는 경쟁하지 않는다** — 레이어가 다르다. 에이전트가 외부 도구/데이터에 접근하는 문제(MCP)와 에이전트가 다른 에이전트에게 태스크를 위임하는 문제(A2A)를 구분해야 한다. 둘을 동시에 쓰는 것이 정상이다.

4. **거버넌스 없는 자동화는 부채다.** Cursor Automations가 프로덕션에서 수백 건의 자동화를 실행한다면, 각 자동화의 권한 범위, 종료 조건, 에스컬레이션 정책이 명시적으로 설계되어야 한다. 이것을 나중으로 미루는 것은 기술 부채가 아닌 거버넌스 부채다.

5. **프로토콜 선택보다 레이어 설계가 먼저다.** 어떤 프로토콜을 쓸지 결정하기 전에, 에이전트 시스템의 어떤 레이어에서 어떤 통신 문제를 해결해야 하는지를 명확히 해야 한다.
