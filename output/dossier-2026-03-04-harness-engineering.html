AI 에이전트를 신뢰할 수 있게 만드는 구조의 설계 원칙

에이전트의 성능은 모델이 아니라 하네스가 결정한다 — 능력의 임계점을 넘은 순간, 엔지니어링의 전쟁은 모델 선택이 아니라 신뢰의 아키텍처 설계로 이동한다.

심층 탐구 1: 하네스의 해부학 — 5개 레이어

무슨 일이 일어났는가

2026년 2월, LangChain 팀이 실증 데이터를 공개했다. 코딩 에이전트를 Terminal Bench 2.0에서 52.8%에서 66.5%로 개선했는데, 모델은 전혀 바꾸지 않았다. 변경한 것은 하네스뿐이었다.

같은 달, Vercel 팀은 에이전트에 제공하는 도구를 15개에서 2개(bash + SQL)로 줄였더니 정확도가 100%로 올라가고 속도는 3.5배 빨라졌다고 발표했다.

두 케이스가 가리키는 방향은 동일하다: 하네스가 에이전트의 성능을 결정한다.

왜 이것이 흥미로운가

직관에 반하기 때문이다. 우리는 AI 시스템을 이야기할 때 보통 "어느 모델이 더 좋은가"를 먼저 묻는다. 하지만 실제 현장에서 쌓이는 데이터는 다른 질문을 가리킨다: "모델을 둘러싼 구조가 얼마나 잘 설계되어 있는가?"

Evangelos Pappas가 제시한 비유가 명쾌하다:

Model = CPU (연산 능력) Harness = OS (자원 관리, 표준 인터페이스, 태스크 조율) Context window = RAM (제한된 작업 메모리) Agent = Application (그 위에서 실행되는 사용자 로직)

스마트폰 시대에 프로세서는 빠르게 commodity화되었다. 가치는 iOS/Android가 가져갔다. GPU raw FLOPS가 commodity화되자, 가치는 CUDA/PyTorch 스택으로 이동했다. 지금 에이전트 시대가 열리면서, 비슷한 이동이 시작되고 있다. 모델을 논쟁하는 것은 어느 CPU가 더 빠른지 논쟁하는 것과 같다 — 중요하지만, 결정적이지 않다.

단, 조건이 있다. 능력 임계점(Capability Floor) 아래에서는 하네스가 모델 한계를 보완할 수 없다. 임계점의 실용적 테스트: "10개의 대표 태스크에서 실패가 추론 문제인가, 오케스트레이션 문제인가?" 추론 문제라면 모델을, 오케스트레이션 문제라면 하네스를 고쳐야 한다.

실제로 어떻게 작동하는가 — 하네스의 5개 레이어

좋은 하네스는 5개의 레이어로 구성된다.

Layer 1: 컨텍스트 엔지니어링 (무엇을 보여줄 것인가)

"컨텍스트 엔지니어링"이란 Tobi Lütke(Shopify CEO)가 2025년에 제시한 개념이다:

"태스크가 LLM에 의해 해결 가능한 수준이 되도록 모든 컨텍스트를 제공하는 기술(art)."

Andrej Karpathy는 이를 더 정밀하게 표현했다:

"컨텍스트 창을 다음 단계를 위해 딱 맞는 정보로 채우는 섬세한 기술과 과학."

핵심 통찰: 컨텍스트 창이 200K 토큰이거나 2M 토큰이거나 — 창이 채워지기 훨씬 전부터 품질이 저하된다. 크기가 아니라 무엇을 넣느냐가 결정한다.

컨텍스트에 포함해야 할 것:

  • 현재 태스크의 목표와 명시적 제약
  • 의미적으로 관련된 코드/문서 (전체가 아니라 선택적으로)
  • 이전 스텝의 결과 (요약 형태, 원문이 아니라)
  • 실패 이력 (어떤 접근법이 작동하지 않았는가)
  • 환경 정보 (디렉토리 구조, 사용 가능한 도구 목록)
  • 시간/비용 예산 경고

컨텍스트에서 빼야 할 것:

  • 관련 없는 대화 이력 전체
  • 이미 완료된 중간 단계의 상세 내용
  • 중복 정보

Manus 팀의 실전 패턴: raw → compressed → summarized 3단계 압축 계층. KV-cache 최적화와 결합해 50번의 도구 호출 태스크를 $3→$0.30/MTok로 10배 비용 절감 달성.

Layer 2: 도구 설계 (무엇을 할 수 있게 할 것인가)

Vercel 케이스의 역설: 도구를 줄이면 성능이 올라간다.

왜인가? 전문화된 도구는 의사결정 병목을 만든다. 에이전트가 "이 케이스에 어느 도구를 써야 하는가"를 계속 결정해야 하고, 이 결정 자체가 에러를 낳는다. 반면 bash + SQL 같은 범용 인터페이스는 에이전트의 학습 데이터와 정렬이 잘 되어 있어 예측 가능하게 사용된다.

좋은 도구 설계 원칙:

  1. 범용 > 전용: 에이전트 학습 데이터와 정렬된 인터페이스를 먼저 선택
  2. 에러 메시지 = 교육: 도구의 에러 메시지를 remediation instruction으로 설계 (Stripe의 접근법 — 린터 메시지가 동시에 수정 안내)
  3. MCP 표준화: Stripe는 400+ 내부 도구를 MCP 서버 "Toolshed"로 노출. 에이전트가 도구를 탐색하는 방식을 표준화
  4. 도구당 단일 책임: 도구가 너무 많은 것을 하면 에이전트가 예측 불가능하게 사용

Mitchell Hashimoto의 실천: "에이전트가 접근할 수 있는 도구 목록과 사용 시나리오를 AGENTS.md에 명시. 에이전트가 잘못 사용하면 즉시 AGENTS.md를 업데이트한다."

Layer 3: 아키텍처 가드레일 (무엇을 하지 못하게 할 것인가)

OpenAI 내부 경험에서 발견한 역설: 솔루션 공간을 제한할수록 에이전트 신뢰성이 올라간다.

구체적으로:

  • 레이어드 아키텍처와 엄격한 의존성 방향 제약 적용
  • AI가 생성한 커스텀 린터로 아키텍처 위반 자동 차단
  • 의존성 그래프를 구조적 테스트로 지속 검증
  • "가비지 컬렉션 에이전트"가 주기적으로 문서 불일치와 아키텍처 위반을 탐지

이것이 이상하게 느껴지는 이유: 일반적으로 에이전트에게 "자유롭게 해결하게" 해야 한다고 생각한다. 하지만 에이전트가 솔루션 공간 전체를 탐색하면 예측 불가능한 결과를 낸다. 제약이 탐색 공간을 줄이고, 검증이 쉬워지고, 에이전트가 "올바른 방향"을 유지하게 만든다.

AGENTS.md: AI 에이전트를 위한 살아있는 README. 에이전트가 매 세션 시작 시 자동으로 읽는다. 에이전트가 실수할 때마다 업데이트하여 피드백 루프를 닫는다. 이 파일의 크기가 하네스의 경험치를 나타낸다.

Layer 4: 피드백 루프 (어떻게 배우게 할 것인가)

Mitchell Hashimoto의 핵심 원칙:

"에이전트가 실수를 할 때마다, 그 실수를 다시 하지 못하게 만드는 구조적 해결책을 만든다."

단순하게 들리지만, 실천은 어렵다. 어디에 그 해결책을 담는가?

피드백 루프 메커니즘 계층:

메커니즘 적용 위치 강도
AGENTS.md 업데이트 명시적 지식 낮음 (에이전트가 무시 가능)
도구 에러 메시지 개선 암묵적 교정 중간
컨텍스트 템플릿 업데이트 구조적 guidance 중간
린터 규칙 추가 기계적 강제 높음 (항상 적용)
CI 파이프라인 테스트 게이트 강제 높음 (통과 없이 머지 불가)

LangChain의 "Trace 분석" 패턴: 에이전트 실패를 대규모로 분석해서 패턴을 발견 → 발견된 패턴을 하네스에 반영. 이 과정 자체를 Agent Skill로 자동화. ML의 boosting 알고리즘과 구조적으로 동일 — 이전 실패에 집중해서 반복 개선.

Layer 5: 평가 시스템 (어떻게 알 수 있는가)

Braintrust 연구의 충격적인 수치:

에이전트가 단일 실행에서 60% 성공률을 보이더라도, 8회 반복 일관성 테스트에서 25%로 떨어진다.

이것이 의미하는 바: LLM eval과 에이전트 eval은 근본적으로 다르다. 에이전트는 멀티스텝 워크플로우이므로, 초기 단계의 에러가 이후 모든 단계를 오염시킨다. "한 번 성공"은 "신뢰할 수 있다"는 뜻이 아니다.

에이전트 eval의 3가지 필수 차원:

  1. Task Completion Rate — 태스크를 실제로 완수했는가 (단순 완료율이 아니라 올바른 완료)
  2. Trajectory Quality — 올바른 도구를 올바른 순서로, 최소한의 스텝으로 사용했는가
  3. Cost Efficiency — 같은 결과를 최소한의 토큰/API 호출로 달성했는가 (동일 성능에서 50× 비용 차이가 관측됨)

실용적 eval 시작점: "10개의 대표 태스크를 실행하라. 실패가 추론 문제라면 모델을 바꿔야 한다. 오케스트레이션 문제라면 하네스를 고쳐야 한다."

심층 탐구 2: 메타 하네스 — 에이전트를 만드는 에이전트

무슨 일이 일어났는가

2026년 2월, arxiv에 "AOrchestra: Automating Sub-Agent Creation for Agentic Orchestration" 논문이 게재되었다 (arXiv:2602.03786). 핵심 기여: 에이전트를 동적으로 생성하기 위한 통합 추상화.

에이전트를 4-튜플로 정의한다:

Agent = ⟨Instruction, Context, Tools, Model⟩

중앙 오케스트레이터가 태스크를 받으면:

  1. 태스크에 관련된 컨텍스트를 큐레이팅
  2. 적절한 도구와 모델을 선택
  3. 그 순간의 태스크를 위한 전문화된 에이전트를 즉석에서 생성
  4. 실행을 위임, 결과를 수집

왜 이것이 흥미로운가

"AI 에이전트를 만드는 AI 에이전트"는 단순히 에이전트가 에이전트를 spawn하는 것이 아니다. 그것은 메타 하네스 문제다: 하네스를 만드는 하네스를 어떻게 설계할 것인가?

여기서 재귀적 어려움이 등장한다:

  • 일반 하네스의 문제: 에이전트가 실수하지 않도록 어떻게 구조화할 것인가
  • 메타 하네스의 문제: 에이전트가 만드는 에이전트 역시 실수하지 않도록 어떻게 구조화할 것인가 — 그리고 그 에러가 부모 에이전트에게 어떻게 전파되는가

첫 번째 하네스의 실수는 고치면 된다. 하지만 메타 하네스의 실수는 그것이 생성한 모든 자식 하네스에 복제된다.

핵심 설계 결정 3가지

결정 1: 정적 하네스 vs 동적 하네스

정적 하네스: 모든 에이전트가 같은 도구/컨텍스트/프롬프트 구조를 사용. 예측 가능하고 디버깅이 쉽다. 하지만 태스크 다양성에 비효율적.

동적 하네스 (AOrchestra 방식): 태스크마다 에이전트를 즉석 생성. 더 효율적이지만 예측 불가능성이 높다. 4-튜플의 각 구성요소를 누가 어떤 기준으로 결정하는가가 핵심이다.

실무적 권고: 신뢰도가 검증된 태스크 유형에서 동적으로 시작. 핵심 인프라와 보안 관련 에이전트는 정적으로 유지. "코어는 정적, 엣지는 동적."

결정 2: 신뢰 전파 (Trust Propagation)

에이전트 A가 에이전트 B를 만들고, B가 C를 만들 때 — 신뢰와 권한이 어떻게 흐르는가?

현재 best practice (아직 표준화 초기):

  • 권한 상속 제한: 자식 에이전트는 부모보다 넓은 권한을 가질 수 없다
  • 출력 검증 게이트: 자식 에이전트의 출력은 부모 하네스의 가드레일을 통과해야 한다
  • 재귀 깊이 제한: 무한 에이전트 생성 방지 (보통 3-4 레벨)
  • 에러 버블업: 자식 에이전트 오류는 명시적으로 부모에게 전파, 조용히 삼키지 않는다

결정 3: 하네스 회귀 방지 (Harness Regression Prevention)

에이전트가 동적으로 하네스를 생성할 때 — 생성된 하네스가 올바른가를 어떻게 검증하는가?

두 가지 접근:

  1. 메타 하네스 평가: 생성된 하네스를 알려진 골든 태스크 세트로 즉시 테스트
  2. 보수적 초기화: 검증된 하네스 템플릿을 기반으로 생성, 변형 범위 제한

Hashimoto 원칙의 메타 레벨 확장: "에이전트가 만든 하네스가 실수하면, 그 하네스의 생성 로직 자체를 고쳐야 한다."

창업자 관점의 함의

에이전트를 만드는 에이전트를 파는 스타트업에게 이것은 제품 정의 질문을 바꾼다:

당신의 제품은 에이전트인가, 메타 하네스인가?

만약 당신이 에이전트를 만드는 에이전트를 판매한다면 — 고객에게 판매하는 것은 결국 "하네스 생성의 품질"이다. 생성된 에이전트가 얼마나 신뢰할 수 있는가, 얼마나 고객의 특수 컨텍스트에 적응하는가.

이것은 방어 가능한 해자(moat)가 될 수 있다:

  • 축적된 에러→수정 이력 (Hashimoto 원칙의 데이터 축적)
  • 도메인별 검증된 (I, C, T, M) 튜플 라이브러리
  • 메타 하네스가 생성한 하네스의 품질을 평가하는 자체 eval 시스템

심층 탐구 3: 5가지 하네스 실패 모드와 방어 패턴

프로덕션에서 반복적으로 관찰되는 실패 모드들이다.

Failure Mode 1: Doom Loop (무한 수정 루프)

증상: 에이전트가 작동하지 않는 접근법을 반복적으로 시도. 같은 에러가 루프. 진행이 없다.

원인: 에이전트는 "첫 번째 그럴듯한 해결책"을 찾도록 최적화되어 있다. 그 해결책이 실패하면, 대안 탐색보다 같은 접근법을 미세 수정하려는 경향이 있다.

방어 패턴:

  • 접근법 블랙리스트: 특정 에러가 N회 발생하면 해당 접근법을 금지 목록에 추가하고 컨텍스트에 명시
  • 루프 감지 미들웨어: 연속 N번의 동일 도구 호출 패턴 감지 → 강제 체크포인트
  • 추론 샌드위치 (Reasoning Sandwich): Planning ← MAX 추론 → Implementation ← MEDIUM → Verification ← MAX. 에이전트가 "생각 없이 실행"하는 경향을 예산으로 교정

Failure Mode 2: Context Pollution (컨텍스트 오염)

증상: 단계가 진행될수록 에이전트가 일관성을 잃는다. 초기 목표를 잊는다. 모순된 결정을 내린다.

원인: 컨텍스트 창이 관련/무관한 정보로 누적된다. "lost-in-the-middle" 현상 — 중간 정보는 컨텍스트 창의 어딘데 있든 모델이 잘 활용하지 못한다.

방어 패턴:

  • 주기적 컨텍스트 압축: raw → compressed → summarized 3단계 계층으로 누적 방지
  • 목표 반복 (Goal Recitation): 각 스텝 시작 시 원래 목표를 컨텍스트 앞부분에 재명시
  • Todo-list 상태 관리: 완료/미완료/진행중 태스크를 컨텍스트 말미에 항상 포함

Failure Mode 3: Tool Proliferation (도구 과잉)

증상: 에이전트가 어느 도구를 써야 할지 결정하는 데 과도한 토큰 소비. 비효율적 도구 선택. 비용 급증.

원인: 사용 가능한 도구가 많을수록 에이전트의 의사결정 부하가 증가한다.

방어 패턴:

  • 도구 계층화: 기본 도구 세트 (범용, 항상 사용 가능) + 전문 도구 세트 (태스크 유형별 로드)
  • 최소 시작: bash + SQL로 시작해서 실제로 필요해질 때만 전문 도구 추가
  • 도구 사용 트레이싱: 어떤 도구가 얼마나 자주 사용되는지 모니터링. 거의 사용되지 않는 도구는 제거 후보

Failure Mode 4: Over-Specification (과잉 명세)

증상: 하네스가 너무 상세하게 에이전트에게 "어떻게"를 지시한다. 에이전트가 새로운 상황에 유연하게 대응하지 못한다. 모델 업그레이드 시 하네스가 깨진다.

원인: 현재 모델의 한계를 하네스로 패치하다 보면, 그 패치가 미래 모델의 능력을 오히려 제한한다.

방어 패턴 — Bitter Lesson의 하네스 버전:

"나중에 지울 수 있는 하네스를 만들어라." — Evangelos Pappas

  • 각 하네스 구성요소에 "이것이 왜 필요한가"와 "어떤 모델 한계를 보완하는가"를 주석으로 명시
  • 모델 업데이트 시 패치 리스트를 검토하고 더 이상 필요 없는 것은 삭제
  • 하네스는 모델 능력이 성장하면 단순해져야 한다, 복잡해지면 안 된다

Failure Mode 5: Verification Vacuum (검증 공백)

증상: 에이전트가 실제로 성공하지 않았는데 성공을 선언한다. 테스트를 건너뛴다. 첫 번째 그럴듯한 결과를 최종으로 제출한다.

원인: LangChain 연구에서 발견한 것처럼, 에이전트는 "첫 번째 그럴듯한 해결책에서 멈추도록" 편향되어 있다. 검증 스텝이 명시적으로 요구되지 않으면 스킵된다.

방어 패턴:

  • Self-verification 강제: 시스템 프롬프트에 "항상 자신의 결과를 테스트하고 실제로 통과하는지 확인하라"를 명시
  • 수락 기준 사전 정의: 태스크 분해 단계에서 "이것이 완료되었다는 것을 어떻게 알 수 있는가"를 먼저 정의 (인간 승인 게이트)
  • CI 통합: 에이전트 출력을 자동 테스트 파이프라인에 통과시키기 전까지 "완료" 처리 안 함

이번 도시에 개념 도구 (Conceptual Toolkit)

하네스 방정식 (Harness Equation) 성능 ∝ (컨텍스트 품질 × 도구 적절성 × 피드백 밀도) / (에이전트 자유도). "모든 것을 허용하는 하네스는 하네스가 아니다." 자유도를 높이면 탐색력은 올라가지만 신뢰성은 떨어진다.

추론 샌드위치 (Reasoning Sandwich) LangChain이 발견한 성능 패턴: Planning에 최고 추론 예산 → Implementation에 중간 → Verification에 다시 최고. 에이전트가 "생각 없이 실행"하는 경향을 추론 예산으로 교정.

Hashimoto의 원칙 "에이전트가 실수를 할 때마다, 그 실수를 다시 하지 못하게 만드는 구조적 해결책을 만든다." — 하네스를 살아있는 시스템으로 대하는 접근. 실수가 하네스 개선의 입력값이 된다. 이 원칙을 꾸준히 적용하면 AGENTS.md의 크기가 경험치의 지표가 된다.

Bitter Lesson의 하네스 버전 "나중에 지울 수 있는 하네스를 만들어라." — 현재 모델의 한계를 영속적인 제약으로 만들지 마라. 하네스는 모델 능력이 성장하면 단순해져야 한다.

능력 임계점 (Capability Floor) 하네스 개선이 효과를 내는 전제 조건. "대표 태스크 10개에서 실패가 추론 문제인가, 오케스트레이션 문제인가?" 추론 문제면 모델을, 오케스트레이션 문제면 하네스를. 이 구분이 엔지니어링 자원 배분을 결정한다.

에이전트 튜플 (Agent Tuple) Agent = ⟨Instruction, Context, Tools, Model⟩ — AOrchestra의 추상화. 에이전트를 동적으로 생성할 때, 이 4가지 구성요소를 독립적으로 조합하고 교체할 수 있으면 메타 하네스가 훨씬 유연해진다. "컴포저블한 에이전트"의 기반 단위.

컨텍스트 압축 계층 (Context Compression Hierarchy) raw → compressed → summarized. 장시간 실행 에이전트에서 컨텍스트 오염을 방지하는 3단계. Manus가 $3→$0.30/MTok 비용 절감에 사용한 패턴.

시그널 간 연결

이번 도시에를 관통하는 하나의 패턴: "더 적게"가 "더 많이"를 이긴다.

  • Vercel: 도구 15개 → 2개, 정확도 100%
  • Manus: 4번의 완전 재구현, 결국 더 단순한 구조로 수렴
  • OpenAI: 아키텍처를 엄격하게 제한할수록 에이전트 신뢰성 향상
  • LangChain: 추론 예산을 아껴 Planning과 Verification에 집중 투자

이것은 이전에 다룬 적응적 연산(Adaptive Computation) — "모든 것에 같은 양을 쓰지 마라" — 과 같은 방향이다. 모델 레이어에서의 원칙이 하네스 레이어에서 동일하게 재현된다. 어텐션이 75% O(n)으로 처리하듯, 하네스도 대부분의 자유도를 제한하고 중요한 곳에만 비용을 집중한다.

또 하나의 연결: 이전 도시에에서 다룬 검증 공백(Verification Vacuum) — AI Fluency Index에서 발견한 "AI가 잘 할수록 인간이 덜 확인한다" — 이 에이전트 레이어에서 동일한 구조로 재현된다. 에이전트도 그럴듯한 출력을 낼수록 자체 검증을 스킵한다. 인간-AI 관계에서 발견된 패턴이, AI-AI 관계에서도 동일하게 나타난다.

열린 질문들

  1. Hashimoto 원칙의 한계: 실수를 반복적으로 구조로 방지하다 보면, 하네스가 특수 케이스의 집합체가 된다. 언제 하네스를 단순화하는 것이 올바른 선택인가? 하네스의 "기술 부채"는 어떻게 관리하는가?
  1. 메타 하네스의 신뢰 문제: 에이전트가 생성한 하네스의 품질을 어떻게 지속적으로 검증하는가? 첫 번째 생성된 하네스가 잘 작동하더라도, 그것이 만든 에이전트가 만든 하네스는? 재귀적 신뢰의 한계는 어디인가?
  1. 모델 능력 임계점의 이동: 6개월 전 작성한 AGENTS.md가 오늘도 유효한가? 현재 모델의 한계를 보완하는 하네스 컴포넌트가, 모델이 업그레이드되면 오히려 성능을 제한하지 않는가?
  1. 평가의 비결정론적 문제: 에이전트 성능이 8회 실행에서 60%→25%로 떨어진다면, "충분히 좋다"는 기준은 어떻게 설정하는가? 비결정론적 시스템의 프로덕션 배포 기준은?
  1. 창업자의 질문: 에이전트를 만드는 에이전트를 판매할 때, 고객은 무엇을 사는 것인가 — 에이전트의 능력인가, 하네스의 신뢰성인가? 이 차이가 제품 설계와 영업 방식에 어떤 영향을 미치는가?

이번 도시에 학습 포인트

  1. 하네스가 모델보다 먼저다: 에이전트 성능이 낮을 때, 모델을 바꾸기 전에 하네스를 먼저 개선하라. LangChain 데이터가 증명한다: 같은 모델, 다른 하네스로 +13.7%p.
  1. 더 적은 도구가 더 많은 성능을 낸다: 도구를 추가하기 전에 현재 도구를 제거해보는 실험을 하라. Vercel의 역설이 당신의 시스템에서도 성립할 가능성이 높다.
  1. AGENTS.md는 살아있는 시스템이다: 에이전트가 에러를 낼 때마다 AGENTS.md가 더 풍부해져야 한다. "에이전트가 실수 → AGENTS.md 업데이트"를 팀의 기본 루틴으로 만들어라.
  1. 평가 전략이 달라야 한다: 단일 실행 성공률로 에이전트를 평가하지 마라. Task completion + Trajectory quality + Cost efficiency + Consistency (8회 반복)를 함께 측정하라.
  1. 메타 하네스의 단위는 (I, C, T, M) 튜플이다: 에이전트를 만드는 에이전트를 설계할 때, 이 4가지를 독립적으로 조합하고 검증할 수 있는 구조를 먼저 만들어라. 하나의 튜플이 당신 제품의 "원자(atom)"다.
