## 심층 탐구 1 — "49%는 어떻게 만들어졌는가"

### 무슨 일이 일어났는가

Anthropic이 Claude를 SWE-bench Verified에서 49%를 달성했다고 발표했다. SWE-bench는 실제 GitHub 이슈를 AI가 자율적으로 해결하는 벤치마크다. 1년 전 최고 성능이 4%였다는 걸 감안하면 숫자 자체는 인상적이다.

하지만 숫자보다 흥미로운 건 **어떻게** 달성했느냐다.

### 왜 이것이 흥미로운가

이 성능의 대부분은 모델 자체의 능력 향상이 아니라 **에이전트 루프 설계**에서 왔다. 같은 Claude 모델을 쓰더라도 어떻게 도구를 주고, 어떻게 피드백을 순환시키고, 어떻게 실패를 처리하느냐에 따라 성능이 2-3배 달라진다.

이것은 "더 강한 모델"이 답이 아닐 수 있다는 신호다. **하네스가 곧 능력이다.**

### 실제로 어떻게 작동하는가

전형적인 에이전트 코딩 루프를 단순화하면:

```
이슈 이해 → 코드 탐색 → 가설 수립 → 코드 수정 → 테스트 실행
→ 실패 → 원인 분석 → 수정 → 테스트 → ... → 성공
```

이 루프에서 성능을 가르는 설계 결정들:

1. **컨텍스트 관리** — 파일 전체를 읽을지, 관련 부분만 추출할지. 컨텍스트 창이 넘치면 모델이 길을 잃는다.
2. **실패 신호 해석** — 테스트 실패 메시지를 모델에게 어떻게 보여주느냐. 전체 스택 트레이스 vs 핵심 오류 메시지만.
3. **멈춤 조건** — 언제 "이건 내가 못 한다"고 판단하고 에스컬레이션할지. 무한 루프 방지.
4. **도구 설계** — bash, 파일 읽기, 검색을 어떻게 추상화하느냐. 도구 인터페이스가 모델의 사고 패턴을 제약한다.

비유하자면: 같은 목수에게 망치만 주는 것과, 망치·톱·줄자·도면을 구조적으로 주는 것의 차이다. 도구의 질이 아니라 **도구의 배치와 순서**가 결과를 바꾼다.

### 이것이 바꾸는 것

**AI 빌더에게:** 모델 선택보다 에이전트 루프 설계에 투자할 이유가 생겼다. 같은 API를 쓰더라도 하네스를 어떻게 짜느냐가 경쟁력이다.

**연구자에게:** 벤치마크가 "모델 능력"을 측정하는 게 아니라 "모델 + 하네스 능력"을 측정한다는 걸 인정해야 한다. SWE-bench 순위는 사실 에이전트 프레임워크 순위에 가깝다.

**실무자에게:** "Claude로 코딩 자동화"를 시도했다가 실망한 사람은 모델이 아니라 루프 설계를 의심해볼 필요가 있다.

### 아직 모르는 것

- 49%의 나머지 51%는 왜 실패하는가? 어떤 유형의 이슈가 여전히 어려운가?
- 벤치마크 성능과 실제 업무 적용 성능 사이의 간극은 얼마나 큰가?
- 에이전트 루프 설계의 "모범 사례"는 수렴하고 있는가, 아직 탐색 중인가?

### 더 알고 싶다면

- [SWE-bench 공식 사이트](https://swebench.com) — 벤치마크 구조와 평가 방법론
- Anthropic의 에이전트 코딩 기술 리포트 (공식 블로그)
- Simon Willison의 에이전트 도구 분석 시리즈 (simonwillison.net)

## 심층 탐구 2 — "추론 토큰이 표준이 된다는 것의 의미"

### 무슨 일이 일어났는가

Google이 Gemini 2.5에 "Thinking" 모드를 공식 인터페이스로 출시했다. 사용자가 명시적으로 추론 깊이를 제어할 수 있다. Anthropic의 extended thinking, OpenAI의 o-시리즈에 이어 세 번째 주요 플레이어가 같은 방향으로 수렴했다.

### 왜 이것이 흥미로운가

개별 기업의 제품 발표가 아니다. 이것은 **"추론을 명시적으로 쓰는 것"이 LLM 아키텍처의 표준 패턴으로 수렴하고 있다**는 신호다.

1년 전만 해도 이건 OpenAI의 실험이었다. 지금은 세 개의 독립적인 연구 조직이 동일한 결론에 도달했다. 수렴은 우연이 아니다 — 그 방향이 실제로 작동한다는 증거다.

### 실제로 어떻게 작동하는가

기존 LLM은 입력을 받으면 바로 출력 토큰을 생성했다. 추론 모델은 다르다:

```
입력 → [내부 사고 토큰 생성] → 출력
         (사용자에게 보이지 않음)
```

"내부 사고"는 실제로 토큰이다 — 모델이 "중간 계산"을 언어 공간에서 수행한다. 이것은 마치 수학 문제를 풀 때 "암산"과 "풀이 과정을 써가며 풀기"의 차이와 유사하다. 같은 두뇌라도 후자가 훨씬 복잡한 문제를 풀 수 있다.

**핵심 트레이드오프:** 추론 토큰은 더 많은 컴퓨팅과 시간을 쓴다. 간단한 질문에 추론 모드를 쓰면 비효율적이다. 그래서 "언제 thinking을 켤지"가 새로운 설계 결정이 됐다.

### 이것이 바꾸는 것

**하네스 설계자에게:** 이제 하나의 쿼리에 두 가지 경로가 생겼다 — 빠른 응답이 필요한 작업 vs 깊은 사고가 필요한 작업. 이 라우팅을 어떻게 할지가 하네스 설계의 새로운 레이어가 된다.

**비용 모델:** 토큰 비용이 "입출력 토큰"에서 "입력 + 추론 + 출력 토큰"으로 복잡해진다. 같은 작업이라도 thinking 깊이에 따라 비용이 크게 달라진다.

**표준화의 함의:** 세 플레이어가 수렴했다는 건 이 패턴이 곧 "기본값"이 된다는 뜻이다. 지금 이것을 이해하지 못한 채 하네스를 짜면, 1-2년 후에는 중요한 기능을 놓치게 된다.

### 아직 모르는 것

- 추론 토큰의 내용이 실제로 "생각"인가, 아니면 더 복잡한 무언가인가? (Anthropic의 Interpretability 팀이 이 질문을 연구 중)
- 추론 깊이를 자동으로 조절하는 최적 전략은 무엇인가?
- 추론 토큰이 길어질수록 성능이 선형으로 오르는가, 아니면 diminishing returns가 있는가?

## 이번 주 개념 도구

**하네스 엔지니어링 (Harness Engineering)**
AI 모델 주변에 제어 시스템과 도구를 구축하는 엔지니어링 분야. 핵심 통찰: 같은 모델이라도 하네스가 능력을 결정한다. "더 좋은 모델"보다 "더 좋은 하네스"가 현실적인 레버리지인 경우가 많다.

**추론 예산 (Thinking Budget)**
추론 모델에서 "얼마나 깊이 생각할지"를 제어하는 파라미터. 작업 복잡도에 따라 동적으로 조절하는 것이 최적이다. 새로운 하네스 설계 결정의 차원.

**시간 지평 벤치마크 (Time Horizon Benchmark)**
METR이 도입한 AI 능력 측정 방식. "AI가 인간의 감독 없이 자율적으로 수행할 수 있는 작업의 최대 시간"으로 능력을 정의한다. 단순 정확도 측정보다 실용적 능력을 더 잘 반영한다.

## 시그널 간 연결 — 이번 주의 큰 그림

이번 주 시그널들을 관통하는 주제: **"에이전트 능력의 경계가 소프트웨어로 이동하고 있다"**

모델 그 자체의 파라미터 수나 학습 데이터보다, **어떻게 실행하느냐** — 루프 설계, 추론 깊이 제어, 도구 인터페이스 — 가 실제 성능 차이를 만들기 시작했다.

이것은 "모델 경쟁"에서 "하네스 경쟁"으로의 전환 신호일 수 있다. GPT-4가 나왔을 때 모든 사람이 API를 쓸 수 있었지만, 실제로 잘 활용한 것은 도구와 프롬프트를 잘 설계한 사람들이었다. 지금은 그 설계의 복잡도가 한 단계 올라갔다.

METR의 시간 지평 데이터가 이것과 연결된다 — 능력이 6개월마다 2배가 된다면, 그 능력을 끌어낼 수 있는 하네스를 미리 설계해두는 것이 실용적 우위가 된다.

4. 하네스 엔지니어링이 별도의 직업 역할이 될 것인가?
5. 능력이 계속 빠르게 올라간다면, 지금 만드는 하네스의 유효 수명은 얼마나 되는가?

## 이번 주 학습 포인트

1. **같은 모델, 다른 하네스 → 다른 능력.** 모델 업그레이드 전에 하네스 설계를 의심하라.
2. **추론 토큰은 이제 세 플레이어 모두의 표준이다.** 언제 켜고 끌지를 설계하는 법을 배워야 한다.
3. **벤치마크는 모델이 아니라 모델+하네스를 측정한다.** 순위를 읽을 때 이 맥락을 잊지 말 것.
4. **"얼마나 잘 하는가"보다 "언제 멈출지를 아는가"가 에이전트 신뢰성의 핵심이다.**
